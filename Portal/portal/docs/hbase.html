HBase

Aufgrund der Zielsetzung eine Platform für horizontal skalierbare Web-Anwendungen bereitzustellen, kommt in der TwoSpot-Plattform keine relationale Datenbank zum Einsatz. Anstelle dessen wird auf die Spalten-Basierte Datenbank HBase gesetzt. Diese basiert auf dem verteilten Dateisysteme HDFS, das den Kernbestandteil von Hadoop bildet.

Verwendung von HBase

Im Unterschied zu relationalen Datenbanken stellt HBase keine SQL-Schnittstelle zur Verfügung. Auch ein großteil der aus SQL-Datenbanken bekannten Funktionalität wie Primärschlüsse, Indices, Fremdschlüssel oder Constraints steht bei Verwendung der HBase-Datenbank nicht zur Verfügung.

Der Zugriff auf die HBase-Datenbank erfolgt über JDO (Java Data Objects). JDO ist ein Standard der eine einheitliche Schnittstelle für das Arbeiten mit persistenten Daten beschreibt. Der Vorteil von JDO gegenüber JPA (Java Persistence API), Hibernate oder anderen OR-Mappern liegt in der Unabhängigkeit zu der zugrundeliegenden Speichertechnologie. Diese Eigenschaft wird auch als datastore-agnostic bezeichnet und bedeutet vereinfacht ausgedrückt, dass nicht nur relationale Datenbanken sondern jede beliebige Speicherlösung eingesetzt werden können. Beispiele hierfür sind Objektorientierte-, Dokumentenbasierte- oder auch Spalten-Basierte Datenbanken. Dementsprechend lässt sich JDO auch mit der HBase-Datenbank einsetzten.

JDO in TwoSpot

Die TwoSpot-Plattform greift für die JDO-Implementierung auf das DataNucleus-Framework zurück. Nachfolgend wird anhand eines einfachen Beispiels (Kommentarfunktion eines Blogs) gezeigt wie Daten in die Datenbank geschrieben und wieder gelesen werden können.

Im ersten Schritt muss die Struktur der zu speichernden Daten beschrieben werden. Dies erfolgt durch die Definition einer Klasse mit Attributen. Nachfolgend ist eine Klasse dargestellt die einen Kommentar mit einem Namen enthält: 

public class BlogEntry
{
	private String username; 
	
	private String message;

	public String getUsername()
	{
		return username;
	}

	public void setUsername(String username)
	{
		this.username = username;
	}

	public String getMessage()
	{
		return message;
	}

	public void setMessage(String message)
	{
		this.message = message;
	}
}

Anschließend müssen die Klasse und die zu persistierneden Attribute über Annotations definiert werden. Die Klasse wird dabei mit @PersistenceCapable annotiert. Darüber erhält JDO die Information, dass es sich um eine persistente Klasse handelt. Alle persistenten Attribute werden mit @Persistent annotiert. Diese Attribute werden von JDO gespeichert. Es ist darüber hinauß sinnvoll nicht persistente Attribute mit @NotPersistent zu annotieren.

Um persistente Objekte der Klasse eindeutig identifizieren zu können muss zusäthlich ein Schlüsselattribut angelegt werden. Dieses muss in TwoSpot immer den Typ Key haben. Um den Schlüssel automatisch mit eindeutigen Werten zu befüllen stellt TwoSpot einen eigenen Generator zur Verfügung. Dieser wird in der @Persistent-Annotation über das Parameter customValuStrategy="keygen" bestimmt.

Die annotierte BlogEntry-Klasse mit dem Schlüssel sieht damit wie folgt aus:

// .. other imports
import org.prot.storage.Key;

@PersistenceCapable
public class BlogEntry
{
	@Persistent(customValueStrategy = "keygen")
	@PrimaryKey
	private Key key;

	@Persistent
	private String username; 
	
	@Persistent
	private String message;
	
	@Persistent
	private int number;

	public String getUsername()
	{
		return username;
	}

	public void setUsername(String username)
	{
		this.username = username;
	}

	public String getMessage()
	{
		return message;
	}

	public void setMessage(String message)
	{
		this.message = message;
	}

	public int getNumber()
	{
		return number;
	}

	public void setNumber(int number)
	{
		this.number = number;
	}
}

Um Instanzen der BlogEntry-Klasse zu speichern muss eine PersisteceManagerFactory erzeugt werden. Über die Factory lassen sich PersistenceManager-Objekte erzeugen. Dies bilden letztendlich die Schnittstelle zur Datenbank.

Das Anlegen einer neuen PersistenceManagerFactory ist relativ "teuer". Aus diesem Grund empfiehlt es sich die Referenz in Form eines Singletons zu speichern. Ein Beispiel dafür ist in der GoGo-Beispielanwendung gegeben.

Der folgende Quellcodeausschnitt erstellt eine Instanz der BlogEntry-Klasse, erstellt ein PersistenceManager-Objekt und speichert das Objet in der Datenbank. 

PersistenceManagerFactory pmf = JDOHelper.getPersistenceManagerFactory("etc/jdoDefault.properties");
PersistenceManager manager = pmf.getPersistenceManager();

try
{
	BlogEntry entry = new BlogEntry();
	entry.setUsername("Kirk");
	entry.setMessage("Beam me up, Scotty!");

	manager.makePersistent(entry);
} catch (Exception e)
{
	// Do some error handling here
}

Nachdem das BlogEntry- Objet gespeichert wurde, soll dieses wieder aus der Datenbank geladen werden. Hierzu wird über den PersistenceManager ein Query erzeugt. Beim Erstellen des Queries empfielt es sich den Objekt-Typ zu übergeben.

Query query = manager.newQuery(BlogEntry.class);

Wird der Query nun direkt ausgeführt, werden alle gespeicherten Objekte vom Typ BlogEntry aus der Datenbank ausgelesen und in Form einer Liste zurückgegeben: 

List result = (List) query.execute();

Die Ergebnisse werden zu einem gewissen Maße zeitlich sortiert ausgegeben. Neuere Objekte befinden sich damit am Anfang der liste, ältere Objekte am Ende. Es ist durchaus möglich, dass Objekte, deren Erstellungsdaten dicht beeinander liegen nicht in der korrekten Reihenfolge ausgegeben werden!

Um Selektion weiter einzuschränken kann dem Query ein Filter übergeben werden. Der folgende Quellcodeausschnitt zeigt einige Beispiele für Filter-Bedingungen:

query.setFilter("number > 25");
query.setFilter("number < 25");
query.setFilter("number >= 25");
query.setFilter("number <= 25");

query.setFilter("number > 25 && number < 30");

Aus Gründen der Performance ist der ODER-Operator nicht implementiert. Alle Filter-Bedingungen die sich nur auf ein Attribut beziehen können mithilfe eines intern verwalteten Indexes sehr schnell verarbeitet werden. Der UND-Operator wird momentan im Speicher ausgeführt und führt somit zu einer erhöhten Ressourcenbelastung im Server. Sollte der Filter zu komplex sein, wird die Verarbeitung vom Server abgebrochen und nur eine Teilmenge der möglichen Ergebnisse zurückgegeben.

DevServer

Der DevServer verwendet dasselbe JDO-Plugin für DataNucleus wie der Server selbst. Lediglich als Speicherbackend kommt im DevServer keine HBase-Datenbank sondern einfache Binärdateien zum Einsatz. Der Funktionsumfang und das Verhalten der Datenbank ist im DevServer und im Produktiv-Server identisch. Lediglich die Performance-Characteristika können sich zwischen den beiden Systemen unterscheiden!
